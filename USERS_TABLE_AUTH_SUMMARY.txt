================================================================================
DATABASE STRUCTURE
================================================================================

Users Table:
  id              UUID PRIMARY KEY
  email           TEXT UNIQUE (used for login)
  username        TEXT UNIQUE
  password_hash   TEXT (bcrypt encrypted)
  role            TEXT ('Admin' or 'Operator')
  is_active       BOOLEAN (can user log in?)
  last_login      TIMESTAMP
  created_at      TIMESTAMP

No Supabase Auth tables needed!

================================================================================
LOGIN PROCESS (TECHNICAL)
================================================================================

1. User submits: { email: "admin@pacs.com", password: "admin123" }

2. POST /api/login calls authenticateUser()

3. authenticateUser() does:
   a) SELECT * FROM users WHERE email = ?
   b) if found: bcrypt.compare(password, password_hash)
   c) if match: return { id, username, email, role, isActive }
   d) if not found: auto-create new user with Operator role

4. Login response includes role:
   {
     "success": true,
     "data": {
       "id": "uuid",
       "email": "admin@pacs.com",
       "role": "Admin",
       "isActive": true
     }
   }

5. Frontend renders menu based on role:
   if (user.role === 'Admin') {
     Show Admin Configuration menu
   }

6. Backend checks role for protected endpoints:
   if (userRole !== 'Admin') {
     return 403 Forbidden
   }

================================================================================
KEY DIFFERENCES FROM SUPABASE AUTH
================================================================================

Supabase Auth Approach:
  ✗ Complex dual system (Auth + Users table)
  ✗ Password managed by Supabase
  ✗ Session management by Supabase
  ✗ Need email confirmation
  ✗ More dependencies
  ✗ Harder to customize

Users Table Approach:
  ✓ Single source of truth (users table)
  ✓ Direct password control
  ✓ Role in same table as credentials
  ✓ Instant login (no confirmation needed)
  ✓ Fewer dependencies
  ✓ Easy to customize
  ✓ Simple queries

================================================================================
SECURITY
================================================================================

Password Security:
  ✓ Bcrypt hashing (10 rounds)
  ✓ No plain-text passwords
  ✓ Unique salt per password
  ✓ Cannot reverse hashing
  ✓ Slow algorithm prevents brute force

Access Security:
  ✓ Role validation on every request
  ✓ User can only access own data
  ✓ Active status prevents login
  ✓ Last login tracking
  ✓ Session via Supabase middleware

COMMON OPERATIONS
================================================================================

Create new user:

Login:
  POST /api/login with email, password

Get current user:
  GET /api/me (returns user with role)

Logout:
  POST /api/logout

Change user role (SQL):
  UPDATE public.users SET role = 'Admin' WHERE email = '...';

Deactivate user:
  UPDATE public.users SET is_active = false WHERE email = '...';

Reset password:
  POST /api/register with same email + new password
  OR manually update password_hash in database
